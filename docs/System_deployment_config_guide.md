# 系统部署与配置需求（持续完善中）

本指南旨在帮助用户在不同硬件环境（分布式集群、带显卡物理机、单容器环境）下，采用统一的脚本和配置方式完成系统部署。无论是本地LLM服务、外部API Key、Embedding/Reranking服务，还是分布式与容器化部署，均可通过本指南实现自动化与标准化。

---

## 1. 部署场景分类

### 1.1 分布式环境部署

- **适用场景**：多台服务器协同工作，服务分布在不同节点。
- **配置要点**：
  - 每台服务器需指定角色（如：LLM Serving、Embedding、Reranking、Frontend等）。
  - 需配置各服务的网络地址与端口。
  - 支持通过统一脚本生成各节点的 `.env` 和 `docker-compose.override.yml`。
  - 可选：使用分布式配置中心（如etcd/consul）管理全局配置。
- **待补充**：
  - 节点自动发现与注册机制
  - 服务健康检查与自动重启策略
  - 分布式日志与监控集成建议

### 1.2 带显卡物理服务器部署

- **适用场景**：单台或多台具备GPU的服务器，主要用于本地LLM推理。
- **配置要点**：
  - 检测并利用本地GPU资源（如NVIDIA CUDA）。
  - LLM Serving、Embedding、Reranking可本地部署，优先使用GPU。
  - 支持本地模型路径配置，或自动下载/挂载模型。
  - 可通过脚本自动检测GPU并生成相应配置。
- **待补充**：
  - 多GPU分配与负载均衡策略
  - GPU驱动与依赖自动检测与安装
  - 模型热更新与回滚机制

### 1.3 单容器环境部署

- **适用场景**：开发测试、轻量级试用，所有服务在单一容器或主机上运行。
- **配置要点**：
  - 使用 `docker-compose` 一键启动所有服务。
  - 支持通过环境变量配置外部LLM API Key。
  - 可选择是否启用本地LLM服务或仅依赖外部API。
  - Embedding/Reranking服务可选本地或云端。
- **待补充**：
  - 容器资源限制与性能调优建议
  - 数据持久化与备份方案
  - 容器日志收集与分析

## 2. 统一部署脚本设计

### 2.1 脚本入口

建议提供 `deploy.sh` 或 `setup.py` 作为统一入口，支持如下参数：

- `--mode`：部署模式选择（distributed/gpu/local）
- `--config`：指定自定义配置文件路径
- `--with-monitor`：是否启用监控与日志收集
- `--auto-detect`：自动检测硬件与环境，推荐配置
- `--dry-run`：仅生成配置文件，不实际部署
- `--update-models`：自动下载或更新所需模型

### 2.2 脚本功能需求（持续补充）

- 自动检测当前硬件环境（CPU/GPU/内存/网络）
- 根据部署模式自动生成 `.env`、`docker-compose.yml`、`docker-compose.override.yml`
- 支持一键启动、停止、重启所有服务
- 检查端口占用、依赖软件（如Docker、NVIDIA驱动）等前置条件
- 日志输出友好，支持详细/简洁模式切换
- 部署完成后自动进行健康检查，并输出访问入口与服务状态
